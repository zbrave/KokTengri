# Level-Up Selection System

> **Status**: Approved
> **Author**: zbrave + game-designer
> **Last Updated**: 2026-04-04
> **Implements Pillar**: Every Element Matters, Build Diversity Over Build Power, Clarity Under Pressure

## Overview

Level-Up Selection is the pause-overlay decision system that appears when the player gains a level and must choose how run progression advances. The system subscribes to `LevelUpEvent` (published by XP & Leveling through Event Bus), freezes active gameplay simulation, and presents three element choices with predictive outcome tooltips generated by Spell Crafting's evaluation API. Its purpose is to make each level-up meaningful, readable, and deterministic: the player sees what each option will do before committing, can use one free re-roll, and always has a valid resolution path even in late-game constrained states (full inventory, full spell slots, or all spells maxed).

The system is strictly a selection UI/UX layer and flow coordinator. It does not calculate crafting recipes, XP curves, or inventory storage logic itself. Instead, it orchestrates inputs and outputs between existing systems (XP & Leveling, Run Manager, Spell Crafting, Element Inventory, Spell Slots, and Event Bus) so the player experience remains transparent and consistent with the approved crafting rules.

## Player Fantasy

The player should feel like a composed Tengri shaman making a high-stakes ritual choice in the middle of chaos, with full clarity about consequences. Level-up moments are designed as short tactical pauses: danger stops, options become readable, and the player can intentionally route toward a new spell, an upgrade, inventory setup, or fallback survival growth.

The emotional target is confidence, not guesswork. A good player should feel rewarded for planning around element inventory and spell slots, while a new player should still understand outcomes through explicit tooltip language. Even in constrained situations (no craft path, full slots, fully maxed spell build), the system should still communicate agency and forward momentum.

## Detailed Rules

### 1) System Role and Responsibility Boundaries

1. Level-Up Selection owns only presentation, input capture, and resolution routing for level-up decisions.
2. Level-Up Selection does not own or duplicate:
   - recipe math (owned by Spell Crafting),
   - XP thresholds and overflow math (owned by XP & Leveling),
   - element slot persistence (owned by Element Inventory),
   - spell instance lifecycle (owned by Spell Slot Manager).
3. Tooltip outcomes are always requested from Spell Crafting evaluation API (single source of truth).
4. Any mismatch between UI prediction and resolved result is considered a blocker defect.

### 2) Trigger, Pause, and Resume Contract

1. Trigger source: `LevelUpEvent` emitted by XP & Leveling through Event Bus.
2. On receipt of a level-up request, Run Manager transitions to paused simulation state.
3. Paused simulation guarantees:
   - enemies are frozen,
   - gameplay timers are frozen,
   - player movement is frozen,
   - no gameplay damage is applied during selection,
   - background world remains visible but dimmed.
4. Overlay close condition: one valid resolution is committed for the current level-up request.
5. Resume condition: after resolution callbacks complete, Run Manager returns to active simulation state.

### 3) Canonical Selection Flow

For each level-up request, execute the following sequence:

1. Player reaches required XP; XP & Leveling publishes `LevelUpEvent`.
2. Run simulation enters paused state.
3. Level-Up Selection overlay appears above dimmed gameplay scene.
4. System generates 3 random element options from pool `{Od, Sub, Yer, Yel, Temür}` using weighted selection.
5. For each option, UI asks Spell Crafting evaluation API for predicted outcome text.
6. Player may consume 1 free re-roll (if unused) to regenerate all 3 options.
7. Player selects one option (or valid fallback option where applicable).
8. Spell Crafting resolves selected element against inventory/spell state.
9. Resolution result is applied (craft, upgrade, inventory add, blocked warning path, discard path, or stat boost path).
10. Overlay closes, pause is removed, gameplay resumes.

### 4) Option Generation Rules

1. Base option count per screen: exactly 3 element cards.
2. Element pool: 5 elements (`Od`, `Sub`, `Yer`, `Yel`, `Temür`).
3. Starting-element bias:
   - hero starting element receives +15% weighted appearance frequency,
   - all non-starting elements use baseline weight.
4. Re-roll behavior:
   - exactly one free re-roll per level-up screen,
   - re-roll replaces all 3 options,
   - re-roll may produce repeated elements and may reproduce prior set,
   - no bad-luck protection in MVP.
5. Sampling uniqueness policy (MVP): duplicates among the three cards are allowed unless disallowed by future tuning.

### 5) Tooltip and Prediction Rules

1. Tooltip text is generated via Spell Crafting `EvaluateSelection(element)` path.
2. Allowed player-facing outcomes for element cards:
   - `If you pick this -> [Spell Name] will be created`
   - `If you pick this -> [Spell Name] Level N -> N+1`
   - `Added to inventory`
   - `Spell slots full`
3. Level-Up Selection must not perform local recipe lookup or infer spell names.
4. Tooltip refresh behavior:
   - initial population when cards are shown,
   - refresh on re-roll,
   - optional highlight refresh on focus/hover/tap preview for accessibility.
5. If evaluation API returns unavailable/invalid response, card is disabled and error state is logged (non-player-facing generic fallback text allowed in dev builds only).

### 6) Fallback and Constraint Resolution Rules

1. Inventory full with no productive path:
   - if Element Inventory is full (`3/3`) and chosen path yields no productive result,
   - show `Discard element` action as a valid resolution option.
2. Spell slots full with blocked craft:
   - if new craft would occur but Spell Slots are full (`6/6`) and no upgrade path exists,
   - present stat boost fallback options.
3. Stat boost fallback set (mutually exclusive):
   - `+3% Max HP`,
   - `+2% Movement Speed`,
   - `+2% Damage`.
4. Very late-game constraint:
   - if all equipped spells are already at max level and no spell progression is possible,
   - do not show element cards; show only stat boost options.
5. Fallback options are resolution-complete actions and close overlay on confirm.

### 7) Queue Handling for Multiple Level-Ups

1. If XP overflow or burst rewards produce multiple level-ups in short succession, requests are queued.
2. Queue processing order is FIFO by event receive order.
3. Only one Level-Up Selection overlay may be active at any time.
4. After resolving one screen:
   - if queue is non-empty, next screen opens immediately while game remains paused,
   - if queue becomes empty, game resumes.
5. Re-roll count is reset per queued screen (1 free each screen).

### 8) Input and UX Rules (Mobile-First)

1. Primary interaction is touch; gamepad and mouse/keyboard are supported where available.
2. Cards and action buttons use large tap targets suitable for mobile ergonomics.
3. Required visible regions on overlay:
   - center: 3 element cards,
   - supporting panel: current element inventory (3-slot view),
   - supporting panel: current spell slots (up to 6),
   - action row: re-roll button with `1 free` indicator,
   - conditional action area: discard and/or stat boost options when applicable.
4. Re-roll CTA state:
   - enabled before use,
   - disabled and visually spent after use.
5. Overlay must maintain readability over active scene using dimming and contrast-compliant text.

### 9) State Machine

| State | Entry Condition | Exit Condition | Behavior |
|---|---|---|---|
| Hidden | No active level-up request | `LevelUpEvent` dequeued | Overlay not visible; game simulation active |
| Showing | Level-up request activated | Card generation complete | Pause active, overlay animates in, initial cards render |
| Evaluating | Cards available for interaction | Selection confirmed OR re-roll pressed | Player browses cards, tooltips visible, optional re-roll |
| Resolving | Player commits choice/fallback action | Resolution callback success | Selected action forwarded to authoritative system and applied |
| Hidden (resume path) | Resolution complete and queue empty | N/A | Overlay closes and game resumes |

Re-roll transition rule:
- `Evaluating -> Showing -> Evaluating` for regenerated options.

Queue transition rule:
- `Resolving -> Showing` if another level-up is queued.

### 10) Interactions with Other Systems

| System | Interface In | Interface Out | Responsibility Split |
|---|---|---|---|
| XP & Leveling | Publishes `LevelUpEvent` | None | XP system decides when level-up occurs; selection system reacts |
| Event Bus | Subscribe `LevelUpEvent` | Optional analytics/UI events | Event Bus routes event flow, no UI logic |
| Run Manager | `PauseRun(reason=LevelUpSelection)` | `ResumeRun()` | Run Manager owns simulation freeze/resume |
| Spell Crafting | `EvaluateSelection(element)` and resolution call | Outcome object (craft/upgrade/add/blocked) | Crafting is authoritative for outcome prediction and resolution |
| Element Inventory | Snapshot for display | Accepts resolved add/consume/discard effects via authoritative systems | Inventory owns slots and persistence |
| Spell Slot Manager | Snapshot for display | Applies craft/upgrade/stat fallback effects | Spell slot system owns spell state |
| Player Movement | Freeze gate during pause | None | Movement system enforces no movement in paused state |
| HUD/Popup UI | Read-only display updates | None | Selection overlay presents data without mutating gameplay directly |

## Formulas

### Element Option Weight

```text
weight(element) = base_weight × (1 + hero_bias)
```

Where:

```text
hero_bias = 0.15 for hero starting element
hero_bias = 0.00 for all other elements
```

| Variable | Type | Range | Source | Description |
|---|---|---|---|---|
| `base_weight` | float | > 0 | `LevelUpSelectionConfigSO` | Baseline per-element sampling weight |
| `hero_bias` | float | 0.00 or 0.15 | Hero setup data | Bonus for hero starting element frequency |
| `weight(element)` | float | > 0 | Calculated at screen generation | Effective draw weight used by option RNG |

Expected behavior:
- Starting element appears more frequently over many level-ups.
- Non-starting elements remain fully available.

### Weighted Draw Probability

```text
P(element_i) = weight_i / Σ(weight_all_elements)
```

| Variable | Type | Range | Source | Description |
|---|---|---|---|---|
| `weight_i` | float | > 0 | Calculated | Effective weight for candidate element |
| `Σ(weight_all_elements)` | float | > 0 | Calculated | Sum of all 5 element weights |
| `P(element_i)` | float | 0..1 | Derived | Probability of drawing element `i` |

MVP constraint:
- No bad-luck protection and no forced diversity constraints.

### Re-Roll Availability

```text
rerolls_remaining_next = max(0, rerolls_remaining_current - reroll_used)
```

| Variable | Type | Range | Source | Description |
|---|---|---|---|---|
| `rerolls_remaining_current` | int | 0..1 | `LevelUpSelectionConfigSO` + runtime state | Remaining free re-rolls on current screen |
| `reroll_used` | int | 0 or 1 | UI action state | Whether player consumed re-roll |
| `rerolls_remaining_next` | int | 0..1 | Derived | Remaining free re-rolls after action |

### Stat Boost Fallback Values

```text
hp_multiplier_next = hp_multiplier_current × 1.03
speed_multiplier_next = speed_multiplier_current × 1.02
damage_multiplier_next = damage_multiplier_current × 1.02
```

| Option | Multiplier | Scope | Selection Rule |
|---|---|---|---|
| Max HP Boost | `×1.03` | Player max HP stat | Mutually exclusive; pick one |
| Movement Speed Boost | `×1.02` | Effective movement speed stat | Mutually exclusive; pick one |
| Damage Boost | `×1.02` | Outgoing damage scalar | Mutually exclusive; pick one |

### Queue Drain Rule

```text
pending_after_resolve = max(0, pending_before_resolve - 1)
```

| Variable | Type | Range | Source | Description |
|---|---|---|---|---|
| `pending_before_resolve` | int | 0..N | Level-up request queue | Screens waiting before current resolution |
| `pending_after_resolve` | int | 0..N | Derived | Screens waiting after current resolution |

Expected behavior:
- Overlay remains active between queued screens until queue reaches zero.

## Edge Cases

| Scenario | Expected Behavior | Rationale |
|---|---|---|
| Multiple level-ups queued from overflow XP | Show screens sequentially (FIFO), one at a time, while game remains paused | Prevents overlay conflicts and preserves deterministic progression order |
| Re-roll produces identical options | Allowed; player receives new random draw with no guarantee of novelty | Keeps MVP RNG simple and transparent |
| All three options result in `Added to inventory` | Still show all options and allow normal selection/re-roll | Avoids hidden filtering and preserves explicit player agency |
| Player AFK on level-up screen | Game remains paused indefinitely until choice is made | Prevents unfair damage; battery concern accepted for MVP |
| Inventory full (`3/3`) and discard is available, but player has reroll left | Player may reroll first, then choose element/discard path | Maintains strategic flexibility before destructive action |
| Spell slots full (`6/6`) and offered elements cannot upgrade existing spells | Present stat boost options as fallback | Avoids dead-end level-up states |
| Very late game: all spells level 5 and full slots | Show only stat boosts (no element cards) | Keeps progression meaningful when spell system is capped |
| Tooltip says `Spell slots full` | Selecting that card resolves through blocked-craft path (no new spell created) | Prediction and resolution must match for trust |
| Pause and resume race during rapid events | Level-up queue takes precedence; resume only when queue empty | Prevents accidental unpause between stacked level-ups |
| Background gameplay event occurs during selection | Event is queued/held by owning systems until run resumes (or ignored per pause policy) | Maintains freeze guarantee and deterministic simulation |

## Dependencies

| System | Direction | Nature of Dependency |
|---|---|---|
| XP & Leveling (via `LevelUpEvent`) | This system depends on XP & Leveling | Receives level-up trigger events to open selection overlay |
| Event Bus | This system depends on Event Bus | Subscribes to level-up events and optional telemetry/event outputs |
| Run Manager | This system depends on Run Manager | Requests pause/resume and enforces single-active overlay lifecycle |
| Spell Crafting | This system depends on Spell Crafting | Uses authoritative evaluation API for tooltip prediction and final resolution |
| Element Inventory | This system depends on Element Inventory | Displays current inventory and enables discard-context decisions |
| Spell Slot Manager | This system depends on Spell Slot Manager | Displays current spell occupancy and upgrade availability context |
| Player Movement | Player Movement depends on this system's pause state | Movement freeze gate active while selection overlay is open |
| HUD/Popup Framework | This system depends on UI framework | Renders cards, tooltips, and fallback action controls |
| Localization System | This system depends on localization | All player-facing tooltip/action text must be localized strings |

Dependency constraints:
- Do not implement recipe logic in Level-Up Selection.
- Do not implement XP gain math or threshold logic in Level-Up Selection.
- Do not mutate inventory directly except through authoritative resolution pathways.

## Tuning Knobs

| Parameter | Current Value | Safe Range | Effect of Increase | Effect of Decrease |
|---|---:|---|---|---|
| `optionsPerLevelUp` | 3 | 2-5 | More control, less randomness pressure | Higher tension, faster decisions |
| `freeRerollsPerScreen` | 1 | 0-3 | More agency and recipe targeting | Stronger RNG pressure |
| `startingElementBias` | +15% | +0% to +30% | Stronger hero identity and opening consistency | Flatter hero-specific shaping |
| `overlayDimAlpha` | 0.55 | 0.35-0.75 | Better foreground focus/readability | More background visibility, less focus |
| `cardTapTargetMinPx` | 96 | 72-144 | Better mobile usability | More compact layout, higher mis-tap risk |
| `tooltipRefreshMode` | OnShow+OnFocus | OnShow only / OnShow+OnFocus | More responsive clarity | Lower update frequency and complexity |
| `discardOptionEnabledWhenFullNoMatch` | true | true/false | Prevents inventory deadlock | Can create dead-end choices |
| `fallbackHpBoostPercent` | 3% | 1%-5% | More survivability scaling | Lower defensive fallback value |
| `fallbackSpeedBoostPercent` | 2% | 1%-4% | Better kiting and positioning | Lower mobility fallback value |
| `fallbackDamageBoostPercent` | 2% | 1%-4% | Faster kill-time scaling | Lower offensive fallback value |
| `queueMaxDepth` | Unbounded (practical runtime bound) | 1-99 | Handles burst XP safely | Lower memory/complexity, risk of dropped requests |
| `allowDuplicateCardsInSingleRoll` | true | true/false | Simpler RNG and transparent randomness | More curated option diversity |

Data source rule:
- All tunable values are config-driven (`LevelUpSelectionConfigSO`/related data), not hardcoded in runtime logic.

## Acceptance Criteria

- [ ] On `LevelUpEvent`, Level-Up Selection overlay opens and Run Manager enters paused state.
- [ ] During overlay, enemies/timers/player movement are frozen and no gameplay damage is applied.
- [ ] Overlay displays exactly 3 element cards in normal mode with localized name + icon + tooltip.
- [ ] Tooltip text for each card is produced by Spell Crafting evaluation API (no duplicated recipe logic in UI).
- [ ] Tooltip outcomes correctly cover: craft, upgrade, add-to-inventory, and blocked-by-full-spell-slots.
- [ ] Player can use exactly 1 free re-roll per screen; re-roll replaces all three cards.
- [ ] Re-roll may return repeated elements or the same combination as before (allowed behavior).
- [ ] Inventory panel (3 slots) and spell slot panel (up to 6) are visible during selection.
- [ ] When inventory is full with no productive path, `Discard element` option is available.
- [ ] When spell slots are full with no upgrade path, fallback stat boost choices are shown.
- [ ] Stat boost fallback options are mutually exclusive and exactly: +3% HP, +2% speed, +2% damage.
- [ ] If all spells are maxed and no spell progression exists, screen shows only stat boosts.
- [ ] If multiple level-ups are queued, screens resolve sequentially without unpausing between them.
- [ ] After a valid resolution and empty queue, overlay closes and gameplay resumes.
- [ ] System is touch-friendly: card and button targets meet configured minimum tap size.
- [ ] No UnityEvents, coroutines, or `FindObjectOfType` are required by this system design.
- [ ] No hardcoded gameplay balance values are required; all values are data-driven.
- [ ] Design remains consistent with Spell Crafting, Element Inventory, Event Bus, and master Kök Tengri spec.
